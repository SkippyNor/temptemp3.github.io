troubleshooting

not working while viewing file locally

You added highlight.js to your blog for prettier code highlighting but find it not working.

Here is the code:


<link rel="stylesheet" href="//cdnjs.cloudflare.com/ajax/libs/highlight.js/9.12.0/styles/default.min.css">
<script src="//cdnjs.cloudflare.com/ajax/libs/highlight.js/9.12.0/highlight.min.js"></script>

The browser should be able to get the resources referenced in the above code, but to your dismay that is not what's happening.

You check the web console of your browser.

ReferenceError: hljs is not defined

You open up another tab in your browser to view-source:. Click. Click. Still nothing.

What now?

You realize that you are viewing the html files locally, which in the case of this blog is a given. Then, you remember that viewing files locally forces the browser to use the 

file:///
protocol; and you realize that due to the protocol-relative URLs while viewing file locally, the browser is trying to access the resource at 
file:///cdnjs.cloudflare.com/.../default.min.css
which to no surprise failed miserably.

How do we fix this?

The easiest way to fix this issue is to simply add https to the protocol-relative URL.


<link rel="stylesheet" href="https://cdnjs.cloudflare.com/ajax/libs/highlight.js/9.12.0/styles/default.min.css">
<script src="https://cdnjs.cloudflare.com/ajax/libs/highlight.js/9.12.0/highlight.min.js"></script>

Now, as you can see highlight.js is working with flying colors.

questions

when does it matter?

When should you care...

when migrating to https

preventing mixed content

When your site is migrating to https, there may be a point where you start requesting resources using protocol-relative URLs. On the other hand, you may choose to opt-out entirely carrying out all requests over https instead.

when security matters

when your site is on http

Requesting a resource using a protocol-relative URL when your site is on http is a risk you don't want to take when security matters.

what does it mean?

What does it mean when...

when someone else does it

When a domain (cdn, etc.) provides a protocol-relative URL to one are more of its resources as seen in the case of the troublshooting: not working while viewing file locally example, it could only mean one thing; that is, the domain supports both protocols and most likely won't try to redirect http to https.

when you do it

When you hand out business cards with protocol-relative URLs, you are sending a message: 1) I have secure content; 2) Even if you do not, that is okay. Not everyone is going make the move; and not everyone was ready when they did. You may redirect http to https.
