how to setup github pages for user

A guide to using GitHub Pages

Skip to fast track; or if you just need to push your html files into a subtree before merging into your master branch, skip to super fast track;

Since the initial release way back in 18 Dec 2008 and the introduction of Origanizations in 30 Jun 2010, there are two types of pages which GitHub allows you to create:

1) User & Organization Pages

2) Project Pages

This guide to using GitHub Pages focuses primarily on User Pages.

* write me *

For further information, see User, Organization, and Project Pages

prerequisites

  1. You have a github account
    • If you don't, go Join GitHub
  2. You already created a repository to host your website
    • If you haven't yet, go Create repository for GitHub Pages
      • Note that in this case you want to follow the steps under User or organization site. Furthermore, the repository name will include your username.
  3. You're done with all of the above
    • go to setup

setup

To get set up, you will need to do some branch work as described in the following workflow.

workflow

  1. If you haven't already, delete all branches with the exception of your default branch, 'master'
  2. Create a new branch off 'master'. For sake of simplicity, let's call the new branch 'test'.
  3. Update your default branch to 'test'
  4. Delete 'master'
  5. Checkout 'test'

update

Update your GitHub Pages by populating the 'html' directory in your repository with html files. Dynamic, static, or straight up vim, how you choose to populate your 'html' directory is up to you. All that matters is that your repository has a 'html' directory with at least 1 html file, preferably index.html.

workflow

  1. update 'html' files
  2. commit and push changes to 'test' 
    
git add .   # add everything
git commit  # commit changes
git push    # push new commits

Now that you have made the changes you want to make public, your are ready to publish. In the case that your Github Pages still shows Error 404: File not found as your top page, you are on first publish; otherwise goto publish iteration

first publish

Publish first update to 'html'

workflow

  1. push subtree 
    
git subtree push --prefix=html/ origin master
  2. update default branch to 'master' (optional)
    • You'll see why you may want to do this later.

publish iteration

Publish updates following first publish

workflow

  1. update 'html' files
  2. commit and push changes to 'test'
  3. push subtree 
    
git subtree push --prefix=html/ origin update

git push using:  origin update
cache for 0be1785234bc21ac2b6f7dc8865fd8181ca00555 already exists!
  4. open a pull request for 'update' comparing with 'master'
  5. merge pull request
  6. delete 'update' (optional)
